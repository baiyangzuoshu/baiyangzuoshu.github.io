&emsp;&emsp;这几年做游戏开发，一个很明显的变化，就是 UI 的生产方式一直在变。十年前我在使用 cocos2d-x，那时候做界面基本就是“手搓”。按钮放哪、背景图怎么对齐、列表怎么排版、不同分辨率怎么适配，都要开发自己写代码处理。UI 不是不能做，而是成本很高，很多时间其实花在了重复劳动上。一个登录界面、一个背包界面，真正复杂的业务逻辑还没开始，光把节点摆好、响应接好，就已经消耗了不少精力。

&emsp;&emsp;到了 2020 年以后，开始更多使用 Cocos Creator，情况就不一样了。Creator 把 UI 搭建从“写代码”变成了“可视化拼接”。节点层级、组件挂载、资源拖拽、布局调整，很多事情都可以直接在编辑器里完成。对开发来说，效率提升是非常明显的。以前要反复改坐标、改锚点、重新编译才能看到效果，现在在编辑器里拖一拖、点一点，立刻就能看到结果。某种意义上，这一代工具把 UI 开发门槛拉低了，也把很多机械性的工作从程序员手里拿走了。

&emsp;&emsp;而现在，AI 又把这件事往前推了一步。它开始尝试“自动拼 UI”。也就是说，开发者不一定非要手动摆节点，也不一定非要自己一个控件一个控件去拖。你只要把需求描述清楚，AI 就有机会先给出一个初版界面，甚至把部分组件结构、样式层级、交互流程一并生成出来。从趋势上看，UI 的生产方式已经从“程序员自己实现”，走到了“工具辅助搭建”，再走到了“AI 参与生成”。

&emsp;&emsp;不过我觉得，这里面最关键的一点不是 AI 会不会画界面，而是你有没有把足够准确的语料喂给它。这里的语料，不只是几句自然语言提示词，更重要的是和项目相关的 API、组件规范、命名规则、交互约束、页面结构，甚至是现有工程里的资源组织方式。AI 不是凭空理解你的项目，它生成出来的 UI 好不好，很大程度取决于你给它的信息是否完整。如果没有这些上下文，AI 能做的往往只是“看起来像那么回事”的 demo，而不是能真正接进项目里的 UI。

&emsp;&emsp;所以我更倾向于把现在这个阶段理解成：AI 不是代替人来做 UI，而是把开发者的工作重心往前移了。以前我们主要在“实现”上花时间，现在更多是在“描述”和“约束”上花时间。你要先把接口定义清楚，把页面状态想明白，把组件边界整理好，再把这些内容交给 AI。谁的工程规范更清晰，谁的 API 设计更稳定，谁就更容易把 AI 用起来。

&emsp;&emsp;从 cocos2d-x 到 Cocos Creator，再到今天 AI 开始拼 UI，我看到的不是某一种工具替代另一种工具，而是 UI 开发正在不断抽象、不断提效。手写代码解决的是“能不能做”，可视化编辑解决的是“能不能更快做”，AI 现在尝试解决的，则是“能不能先帮你做出第一版”。至于最后效果怎么样，还是取决于开发者对业务、工具和工程上下文的理解。
