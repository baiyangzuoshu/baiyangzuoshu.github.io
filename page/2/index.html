C 语言的历史：从 Unix 实验室到现代系统编程

  C 语言是一门很特别的语言。它不算年轻，也不追求语法华丽，但直到今天，操作系统、编译器、数据库、嵌入式设备、运行时和大量底层库，仍然离不开它。理解 C 语言的历史，其实也是理解现代软件工业的一条主线：计算机从实验室走向工程化，软件从机器相关走向可移植，程序员从直接操纵硬件走向构建大型系统。

由来：从 CPL、BCPL 到 B

  C 语言不是凭空出现的。它的血脉可以追溯到 1960 年代的 CPL 和 BCPL。CPL 试图成为一种强大的通用语言，但设计复杂；BCPL 则由 Martin Richards 在 1960 年代后期设计，目标更朴素：简单、紧凑、适合系统编程。

  后来，Ken Thompson 在贝尔实验室基于 BCPL 设计了 B 语言。B 语言已经有了很多后来 C 的影子，但它主要面向早期机器，类型系统很弱。那个时代的硬件资源很紧张，语言必须足够小、足够贴近机器，否则根本跑不起来。

  这就是 C 诞生前的背景：程序员需要一种比汇编高级、但又不像大型语言那样笨重的工具。它要能写操作系统，要能直接处理内存和机器字，又要能让代码在不同机器之间迁移。

诞生：Unix 需要一门新语言

  C 语言真正诞生于贝尔实验室的 Unix 项目。1969 年前后，Ken Thompson、Dennis Ritchie 等人开始开发 Unix。早期 Unix 很多部分用汇编写，这在性能上直接，但在可维护性和移植性上都很困难。

  Dennis Ritchie 在 B 语言基础上继续改进，加入更明确的类型系统、结构体、指针等能力，逐渐形成了 C 语言。大约到 1972 年，C 已经开始被用于 Unix 开发；到 1973 年，Unix 内核的重要部分被 C 重写。这一步非常关键，因为它证明了高级语言不仅能写应用，也能写操作系统。

  C 的成功不只是语法成功，而是和 Unix 绑定在了一起。Unix 需要 C 来获得可移植性，C 也借 Unix 获得真实系统的锤炼。二者互相成就，最终影响了后来的操作系统、网络、编译器和整个开源软件世界。

发展：K&R C 与标准化

  1978 年，Brian Kernighan 和 Dennis Ritchie 出版《The C Programming Language》。这本书后来被称为 K&R，它不仅是 C 语言教材，也在很长时间里承担了事实标准的角色。很多程序员就是通过这本书认识 C 的。

  随着 Unix 传播，C 也快速扩散到高校、研究机构和商业公司。问题随之出现：不同编译器对 C 的实现并不完全一致。如果没有统一标准，代码可移植性就会受到影响。

  于是标准化开始推进。1989 年，ANSI 发布 C 标准，通常称为 ANSI C 或 C89；1990 年，ISO 也发布对应标准，也就是 C90。这个阶段的意义在于，C 从贝尔实验室和 Unix 社区里的实用语言，变成了工业界共同遵守的标准语言。

  之后 C 继续演进。C99 引入了很多现代化能力，比如 // 注释、变长数组、inline、更完整的整数类型等；C11 引入线程、原子操作和更明确的内存模型；C17 主要是修正和维护；目前最新正式发布的 ISO 标准是 ISO/IEC 9899:2024，通常称为 C23。它继续清理历史包袱，也补充了一些更适合现代开发的语言和库特性。

影响：现代语言的共同祖先

  C 的影响远远超出了它本身。C++ 直接从 C 扩展而来，Objective-C、Java、C#、JavaScript、Go、Rust 等语言，都能看到 C 风格语法的影子。大括号、分号、表达式、控制流、函数调用方式，很多都成为后来程序语言的共同习惯。

  更重要的是，C 建立了一套系统编程的思维方式。它让程序员清楚地看到内存、地址、指针、栈、堆、结构体和二进制布局。高级语言可以隐藏这些细节，但很多底层问题最终还是会回到这些概念上。

  这也是为什么学习 C 仍然有价值。哪怕以后主要写 Java、Go、Rust 或 Python，理解 C 都能帮助你理解操作系统、网络、编译、性能和内存问题。C 像是一层底座，不总是在台前，但一直在支撑上层软件。

直到现在：仍在核心位置

  今天，C 语言已经五十多年历史了。它当然有明显缺点：手动内存管理容易出错，指针带来安全风险，未定义行为让调试变复杂，大型工程也需要额外规范约束。很多安全问题，确实和 C/C++ 这类底层语言的内存模型有关。

  但 C 仍然没有被替代。原因很现实：它足够贴近硬件，编译器成熟，运行时负担小，ABI 稳定，生态庞大。Linux 内核、嵌入式系统、数据库核心、图形库、语言解释器、驱动程序，仍然大量使用 C。很多新语言即使自己更安全，也需要和 C ABI 交互，因为 C 已经成为系统软件世界的共同接口。

  所以 C 语言的历史不是一门旧语言被淘汰的历史，而是一门基础语言不断被重新定位的历史。它从 Unix 实验室走来，经历 K&R、ANSI、ISO 标准化，再走到 C23。今天它不再是所有场景的最佳选择，但在需要控制性能、内存、二进制接口和底层资源的地方，C 仍然是绕不开的语言。

  理解 C，也是在理解软件和机器之间那层最直接的关系。

DeepSeek-TUI：在终端里使用 DeepSeek 写代码

  最近我在思考一个更省成本的 AI 编程工作流：让 Codex 负责架构设计、任务拆解和代码审查，让 DeepSeek 负责具体实现。沿着这个思路，我看到了一个项目：DeepSeek-TUI。它的定位很直接：把 DeepSeek 变成一个运行在终端里的 coding agent。

  简单说，DeepSeek-TUI 不是网页聊天工具，而是一个面向本地代码库的终端助手。你可以在项目目录里启动它，让它读取上下文、解释代码、修改文件、运行命令，并通过终端界面和你交互。对习惯命令行开发的人来说，这种方式比来回复制代码更自然。

DeepSeek-TUI 是什么

  DeepSeek-TUI 的入口命令是 deepseek。根据官方 README，它由两个 Rust 二进制组成：一个是调度命令 deepseek，另一个是 TUI 运行时 deepseek-tui。npm 包本身更像安装器和包装器，会下载对应平台的预编译二进制，而不是用 Node.js 实现整个 agent。

  它比较适合几类场景：在终端里让 AI 阅读当前项目；让 AI 按要求修改代码；用一次性命令解释某个函数；在自动化脚本里调用 deepseek exec；或者在已有工作区里恢复历史会话。它也提供 Plan、Agent、YOLO 等模式，分别对应只读规划、默认交互和自动批准工具调用。

使用建议

  DeepSeek-TUI 真正有价值的地方，不只是“能在终端聊天”，而是它能贴近本地项目工作流。但正因为它能改文件、跑命令，使用时更应该给清楚边界。比如让它只修改某几个文件，只实现一个明确功能，先输出方案再动手，最后要求给出测试命令和变更说明。

  我比较推荐的方式是：让 Codex 或自己先做架构拆解，明确模块边界、文件范围和验收标准，然后把具体实现交给 DeepSeek-TUI。完成后再看 git diff，确认它有没有越界修改、重复实现或者破坏原有结构。

  AI 写代码的成本正在下降，但软件维护成本不会自动下降。DeepSeek-TUI 这类工具能提高实现速度，前提是人要把目标、约束和审查做好。把它当成一个终端里的执行工程师，而不是一个可以无限放权的架构负责人，使用体验会更稳定。

AI 写代码后，更需要架构师能力

  这段时间使用 Codex、DeepSeek、Claude 这些 AI 工具写代码后，我越来越明显地感觉到：AI 并没有降低对程序员能力的要求，反而把要求往上提了一层。以前写代码，很多时间花在具体实现上；现在具体实现可以交给 AI，但你必须知道要让它做什么、做到哪里、不能碰哪里。

  换句话说，AI 写代码之后，更需要人具备架构师的能力。

  AI 很擅长执行。你让它写一个接口、补一个组件、改一段逻辑、生成一组测试，它往往能很快给出结果。尤其是在需求边界清楚、上下文完整、约束明确的时候，它的效率非常高。过去半天才能完成的重复性开发，现在可能十几分钟就能推进到可运行状态。

  但问题也在这里。AI 太擅长执行了。如果你没有给它清楚的目标，它也会继续执行；如果你没有告诉它边界，它就可能扩大修改范围；如果你没有说明项目里的分层、模块职责、数据流和异常处理方式，它就会按照自己“看起来合理”的方式实现功能。代码表面上跑通了，实际上却给系统埋下了维护成本。

  很多时候，AI 写出来的问题不是语法错，也不是功能完全不能用，而是架构上不干净。比如本来应该在 service 层处理的逻辑，被它塞进了 controller；本来应该复用已有组件，它重新写了一套类似实现；本来只是改一个小功能，它顺手改了公共工具函数；本来项目有统一错误处理，它又加了一套局部判断。这些问题短期看不明显，长期会让项目越来越乱。

  所以使用 AI 写代码，第一件事不是急着让它开写，而是先划定边界。你要告诉它：这次只改哪些文件，不改哪些文件；新增逻辑放在哪一层；是否允许改公共接口；是否需要兼容旧数据；错误应该如何返回；测试覆盖到什么程度。边界越清楚，AI 越像一个高效工程师；边界越模糊，AI 越像一个精力充沛但方向不稳定的实习生。

  第二件事，是把任务拆成可验证的小块。不要直接说“帮我实现一个完整系统”，而是先让它理解现有结构，再让它实现其中一部分。比如先补数据模型，再补服务逻辑，再补接口，再补测试。每一步都有输入、输出和验收标准。这样做不是为了限制 AI，而是为了让它始终在正确轨道上推进。

  第三件事，是人要做最终判断。AI 可以写代码，但不能替你承担架构责任。它不知道这个项目未来会不会扩展，不知道哪些模块是历史包袱，不知道哪些接口已经被外部依赖，也不知道团队的维护习惯。它只能根据当前上下文推断，而架构判断需要经验、取舍和长期视角。

  以前我们常说，程序员要从“写代码的人”变成“解决问题的人”。AI 出现后，这句话变得更具体了：程序员要从“亲自实现每一行代码的人”，变成“定义问题、拆解系统、设定边界、验收结果的人”。真正拉开差距的，不再只是手速，而是你能不能把一个模糊需求拆成清晰结构。

  AI 会让代码生成越来越便宜，但不会让架构变得便宜。相反，代码越容易生成，架构越重要。因为糟糕的代码过去写得慢，破坏速度有限；现在 AI 可以很快写出大量代码，如果没有人控制方向，系统变坏的速度也会变快。

  所以我现在使用 AI 写代码时，先想清楚目标，再描述边界，再让 AI 实现，最后审查它是否符合项目结构。AI 负责把砖搬得更快，人负责决定墙应该建在哪里。只有这样，AI 才是在提高生产力，而不是在制造新的技术债。

GPT 的三段历史：从语言模型到 ChatGPT 爆发

  读《吴军讲 GPT》时，我印象最深的一点是：ChatGPT 看起来像突然出现的新东西，但它背后其实是一条很长的技术线。吴军把 ChatGPT 称为“被神化的语言模型的衍生品”，这个说法很准确。它让很多人第一次直观感受到机器可以对话、回答问题、写文章，但从技术史看，它不是凭空降临，而是语言模型几十年演进后的结果。

  如果把 GPT 的历史讲清楚，我觉得可以分成三个阶段：最初的语言模型阶段，深度学习推动的发展阶段，以及 ChatGPT 之后的爆发阶段。

一、最初：语言模型从统计开始

  GPT 的根，不在 2022 年，也不在 OpenAI 成立之后，而在更早的自然语言处理研究里。吴军在课程里提到，1972 年，贾里尼克在 IBM 沃森实验室建立了人类第一个语言模型。那个时代的目标不是做一个会聊天的智能体，而是让计算机更好地处理语音识别、机器翻译、文本预测等任务。

  所谓语言模型，最朴素的理解，就是判断一串词出现的可能性。比如看到“我今天早上喝了一杯”，模型要能判断后面接“咖啡”比接“石头”更自然。早期方法主要依赖统计，把大量语料里的词频和共现关系计算出来，再用概率预测下一个词。

  这听起来简单，但意义很大。因为语言第一次被放进了可计算的框架里。机器不再只是查字典、套规则，而是开始从真实语料中学习语言习惯。吴军也提到，他在 1990 年代做语音识别时，和郭进博士较早构建了中文语言模型。后来在贾里尼克和库日普的指导下，又利用 NSF 项目构建第二代语言模型。这说明语言模型不是新概念，它已经走了很久。

  不过这一阶段的语言模型还比较弱。它能帮机器做局部判断，却很难理解长文本，也很难完成复杂问答。它像是给机器装上了一套“语言概率感”，但还没有形成今天这种生成和对话能力。

二、发展：深度学习把语言模型推上台阶

  第二个阶段发生在 2000 年以后，尤其是 2010 年前后。吴军在课程里提到，2010 年，Google 的杰夫·迪恩等人实现了真正实用的深度学习系统，也就是 Google 大脑，并用它构建出第三代语言模型。这是关键转折。

  深度学习改变了语言模型的能力边界。过去模型主要依赖人工设计特征和局部统计，到了神经网络时代，模型可以把词、句子、上下文映射到高维空间里，自动学习更复杂的关系。它不只是知道两个词经常一起出现，还能捕捉语义、语法和上下文关联。

  吴军举过一个例子：2012 年他回到 Google 后启动计算机问答项目，到 2014 年，语言模型已经能够回答一部分复杂问题，也能完成一些写作任务。虽然当时还远没有今天 ChatGPT 的流畅程度，但方向已经很清楚：计算机开始尝试回答“为什么”和“怎么做”。

  随后，2017 年 Transformer 架构出现，注意力机制让模型能够更有效地处理长上下文；2018 年 OpenAI 推出 GPT，2019 年有 GPT-2，2020 年 GPT-3 出现。GPT-3 的参数规模和训练数据大幅增长，生成能力明显跃迁。但吴军提到，他在 2020 年内部讲座中介绍 GPT-3 时，听众并没有太大反应。那时大家更关注大公司的 AI 动态，对 OpenAI 这家小公司并不敏感。

  这说明技术成熟和社会感知并不同步。GPT-3 已经很强，但还没有以普通人容易理解、容易使用的产品形态进入大众视野。发动机已经转起来了，只是还没有装进一辆人人都能上手的车里。

三、爆发：ChatGPT 让语言模型被大众看见

  第三个阶段，就是 2022 年底 ChatGPT 的爆发。为什么同样是语言模型，ChatGPT 一出来就火了？核心原因不是它第一次会生成文字，而是它把能力包装成了最自然的交互方式：对话。

  过去很多 AI 能力藏在搜索、推荐、翻译、语音识别这些产品背后，用户感受到的是功能变好了，却不一定知道背后是语言模型。ChatGPT 不一样，它直接坐在用户面前。你问一句，它答一句；你让它改写、总结、写代码、写公文，它立刻给出结果。普通人第一次感觉到，机器不是只能执行命令，还能参与思考过程。

  这也是它被“神化”的原因。吴军用哥伦布月食的故事提醒大家：当人们不了解现象背后的原理时，就容易把它当成神迹。ChatGPT 的效果确实惊人，但它仍然是语言模型，不是突然产生了人类意义上的智慧。它的底层逻辑依然是大规模语料、模型参数和概率预测，只是规模、结构、训练方法和产品形态都走到了临界点。

  从最初到发展再到爆发，GPT 的历史其实很清楚：第一阶段解决“让机器计算语言”的问题，第二阶段解决“让机器学习复杂语言关系”的问题，第三阶段解决“让普通人直接使用语言模型”的问题。ChatGPT 的伟大，不在于它脱离了语言模型传统，而在于它把几十年的积累变成了一个人人都能感知的产品。

  所以理解 GPT，不能只看热闹。它不是魔法，也不是一夜之间的奇迹。它是语言模型、深度学习、算力、数据、工程和产品交互共同作用的结果。把这三段历史看明白，再看今天的大模型浪潮，就不会只剩下赞叹或恐惧，而能更冷静地判断：哪些是真进步，哪些只是包装；哪些工作会被替代，哪些能力反而会变得更重要。
